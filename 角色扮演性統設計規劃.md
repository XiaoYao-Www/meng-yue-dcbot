# AI TRPG Discord Bot - 系統架構與工程設計檔

## 1. 專案概述 (Project Overview)
本專案旨在開發一個基於 Discord 的自動化 AI TRPG (桌上角色扮演遊戲) 系統。
系統將以 Discord 論壇頻道 (Forum Channel) 作為遊戲的 UI 與互動介面，以 AI (LLM) 作為遊戲主持人 (Game Master)，並嚴格使用關聯式資料庫 (SQLite) 作為全局狀態管理的單一事實來源 (SSOT)。

**核心設計原則：**
- **規則與數值固定：** AI 僅負責「內容與敘事生成」，不負責底層數值運算（如擲骰機率、血量扣除），以防止長期數值膨脹。
- **資料庫驅動 (DB-Driven)：** 所有的狀態變更必須先寫入資料庫，再由資料庫的變更觸發 Discord 貼文/訊息的更新。
- **漸進式資訊揭露 (Progressive Disclosure)：** 利用 Discord 子貼文存放實體 (玩家、怪物、道具) 的文件，隨著探索進度更新貼文內容。

## 2. 核心技術棧 (Tech Stack)
*請 AI 實作時優先採用以下技術：*
- **後端語言：** python (確保型別安全)
- **Discord API：** Discord.py
- **資料庫：** SQLite (儲存為 `.db`，方便管理與轉移)
- **ORM：** Prisma 或 Drizzle ORM (用於簡化資料庫操作與 Schema 管理)
- **AI 整合：** 官方 LLM SDK (如 OpenAI Node SDK 或 Google Gen AI SDK)

---

## 3. 系統架構 (System Architecture)

系統分為四大核心層級：

### 3.1 基礎設施與展示層 (Discord Layer)
- 攔截文字前綴指令 (如 `加入`, `D20`, `使用`)，而非過度依賴斜線指令。
- 負責建立論壇主貼文 (對應一局遊戲)、建立子貼文 (文件化實體、道具等)、發送系統日誌與劇情推進。

### 3.2 狀態與數據層 (Database Layer - SQLite)
- 所有遊戲世界觀、玩家數值、道具歸屬、NPC 狀態的唯一儲存點。
- 遊戲伺服器重啟時，全數從資料庫重建記憶體快取。

### 3.3 邏輯與事件層 (Logic & Event Layer)
- 處理擲骰運算、合法性驗證、數值加減。
- **事件引擎：** 將玩家輸入轉化為標準化 JSON 格式，搭配「事件模板」(探索、戰鬥、對話)，再送交給 LLM，降低 AI 生成的隨機性。

### 3.4 內容生成層 (AI GM Layer)
- 讀取資料庫的「當前狀態摘要」，結合玩家指令生成劇情敘事。
- 負責產生新物件 (如發現新道具)、NPC 對話，並回傳格式化的意圖 (Intent) 供邏輯層更新資料庫。

## 4. 資料庫綱要設計 (Database Schema)

*AI 實作提示：請根據以下概念設計 ORM Schema。*

- **Game (遊戲局)**
  - `id` (PK), `discord_channel_id` (主貼文 ID), `name`, `theme`, `current_stage` (創角/進行中/結束), `world_state_summary` (用於記憶壓縮)
- **Player (玩家角色)**
  - `id` (PK), `game_id` (FK), `discord_user_id`, `name`, `hp`, `san`, `attributes` (JSON), `status` (存活/死亡)
- **Entity (實體：NPC/怪物/道具)**
  - `id` (PK), `game_id` (FK), `discord_thread_id` (對應的 Discord 貼文 ID), `type` (item/npc/monster), `name`, `description`, `owner_id` (FK, 可為 null), `is_known` (boolean, 玩家是否已發現)
- **ActionLog (系統與劇情日誌)**
  - `id` (PK), `game_id` (FK), `timestamp`, `actor_id`, `action_type`, `content`, `result_state`

## 5. 核心工作流 (Core Workflows)

### 5.1 創局與世界生成 (Game Initialization)
1. 使用者觸發創局。
2. 邏輯層呼叫 LLM 產生世界觀、怪物池、道具池。
3. 寫入 DB `Game` 與 `Entity` 資料表。
4. Discord 建立論壇主貼文，並將設定發布於首則訊息。

### 5.2 玩家創角狀態機 (Character Creation)
1. 玩家在主貼文輸入 `加入`。
2. DB 建立 `Player` 初始紀錄，Discord 建立玩家專屬子貼文。
3. 系統在子貼文透過文字引導玩家選擇職業/屬性。
4. 玩家回覆後，更新 DB，並更新玩家專屬子貼文的標題與內容。

### 5.3 行動判定迴圈 (Action Loop)
1. 玩家輸入 `D20 搜尋房間`。
2. **邏輯層攔截：** 從 DB 讀取房間狀態與玩家數值，進行 D20 隨機數生成與規則判定 (成功/失敗)。
3. **AI 生成層：** 將判定結果、當前 DB 狀態送給 LLM：`"玩家擲骰: 成功, 地點: 實驗室. 請描述發現隱藏道具"`。
4. **狀態更新：** LLM 回傳劇情與新物件資料 -> 寫入 DB (建立新道具紀錄)。
5. **UI 同步：** Bot 在 Discord 發布劇情敘事，並自動建立/更新新道具的子貼文，最後在「系統日誌貼文」追加一筆紀錄。

## 6. 實作階段規劃 (Implementation Roadmap)

請 AI 助手依照以下階段進行開發：

- **Phase 1: 基礎架構與資料庫建立**
  - 初始化專案。
  - 設定 SQLite 與 ORM (Prisma/Drizzle)，完成上述 Schema 的 Migration。
  - 建立基本的 Discord Bot 監聽架構 (連線、基礎訊息回應)。
- **Phase 2: 核心遊戲邏輯與 Discord UI 映射**
  - 實作「創建遊戲」流程：能成功在論壇頻道建立主貼文，並寫入 DB(目前已完成論壇創建與第一則貼文創建的基本邏輯，但正規需要更改創建指令，要包含[故事大綱、目標類型(明確目標/隱藏目標)(影響到AI如何設計世界設定與推進劇情發展)
  、最大玩家數等])。
  - 實作「文字指令解析器」(Parser)，能辨識 `加入`, `D20` 等關鍵字。
  - 實作同步機制：DB 資料變更時，自動編輯對應的 Discord 子貼文。
- **Phase 3: AI GM 整合與事件引擎**
  - 串接 LLM API (Deepseek api)。
  - 實作 Prompt Builder：能將 DB 內的當前遊戲狀態序列化為 LLM 可以理解的 Context。
  - 實作 LLM Output Parser：確保 LLM 回傳的結果能拆分為「劇情文本」與「狀態變更指令 (JSON)」。
- **Phase 4: 記憶壓縮機制**
  - 實作背景排程或觸發器：當單局互動超過 N 則時，呼叫 LLM 進行摘要，並更新 DB 的 `world_state_summary`。
