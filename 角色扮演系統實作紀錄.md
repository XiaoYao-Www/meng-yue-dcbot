# AI TRPG 角色扮演系統 — 實作紀錄

> 本文件記錄系統開發過程中的設計想法、困惑、決策取捨與待解決問題，作為了解與決策的參考依據。

---

## Phase 0 — 專案回顧與重構起點

### 背景

之前已經實作了一些 RPG 基礎功能：
- `/rpg_start` — 建立論壇頻道 + 初始貼文 + 5 個標籤
- `加入` 指令 + 硬編碼職業創角（戰士/法師/遊俠）
- `D20` 擲骰 + 簡易 AI GM 迴圈
- 4 張表的 `trpg.db`（games/players/entities/action_logs）

### 問題

使用者指出三個核心問題：

1. **❌ 硬編碼職業** — 戰士/法師/遊俠是寫死的，不符合「AI 動態生成世界」的目標
2. **❌ 資料庫太簡陋** — 沒有 NPC 表、道具表、對話歷史、劇情紀錄等關鍵資料
3. **❌ 世界觀不是 AI 生成的** — 每局遊戲的世界規則應該是 AI 根據故事大綱即時創造的

### 新方向

**核心原則：每局遊戲是一個獨立世界，所有規則、種族、職業、數值都由 AI 在創局時動態生成。**

## Phase 1 — 資料庫設計

### 設計思路

所有遊戲共用同一個 `trpg.db`，以 `game_id` 隔離。一個 SQLite 檔案即可，不需要每局一個檔案。

### 表格結構

| 表 | 用途 | 關鍵欄位 |
|---|------|---------|
| `games` | 遊戲局本體 | world_rules (JSON), world_setting (TEXT) |
| `characters` | 玩家 + NPC 統一存放 | discord_user_id 可 null (NPC), stats (JSON) |
| `items` | 道具 | properties (JSON), owner_id nullable |
| `npcs` | NPC/怪物 | stats (JSON) 獨立表以便查詢 |
| `inventories` | 角色持有道具 | character_id, item_id, quantity |
| `dialogues` | AI 對話歷史 | 完整文字紀錄 |
| `narrative_log` | 劇情紀錄 | action, narrative, state_changes |
| `action_logs` | 系統後端日誌 | 保留原有結構 |

### ✅ 已決定

- 使用單一 `trpg.db` 檔案（非多檔案）
- 繼續使用 `aiosqlite` raw SQL（不使用 ORM）

### ❓ 待確認

- 是否需要 `locations` 表來追蹤玩家當前位置？

## Phase 2 — AI 世界生成引擎

### 設計思路

`/rpg_start` 執行時，在建立論壇頻道之後、回覆成功之前，插入一段非同步 AI 呼叫。

### 生成內容

1. **world_setting** — 世界觀文字描述
2. **world_rules** — JSON 結構，包含：
   - 可用種族列表
   - 可用職業列表 + 初始屬性模板
   - 技能列表
   - 戰鬥/規則說明
3. **初始 NPC/生物池**

### 風險

- ⚠️ AI API 可能逾時或失敗 → 需要 fallback 機制（引導式表單讓創建者手動填寫）
- ⚠️ 生成時間過長 → 需要進度提示而非讓使用者在不知情下等待

### ❓ 待確認

- fallback 模式要怎麼設計？用固定模板還是純文字輸入？

## Phase 3 — AI 驅動創角

### 設計思路

玩家輸入 `加入` 後：
1. 建立角色子貼文
2. 呼叫 AI 生成創角引導（根據該局的世界規則）
3. 玩家與 AI 對話互動完成創角
4. 更新 DB + 子貼文

### 與舊邏輯的差異

- 舊：玩家選「戰士」→ 套用 `{hp:150, san:60, 力量:5}`
- 新：AI 說「在這個世界你可以選擇...」→ 玩家選擇 → AI 設定數值

### ❓ 待確認

- 創角流程是單輪選擇還是多輪對話（例如：先選種族 → 再選職業 → 再分配屬性點）？
  - 目前先實作單輪（AI 一次給選項，玩家選完即完成）
  - 未來可擴充為多輪

## Phase 4 — AI GM 核心迴圈

### 設計思路

`D20 行動描述` → 擲骰 → 送 AI → 解析回應 → 更新 DB → 發佈劇情

### Prompt 結構重大改變

- 不再硬編碼 `max_hp = attrs.get("體質",10)*30`
- 改為從 `world_rules` 動態讀取規則
- 傳入最近 N 條 `narrative_log` 作為上下文
- 傳入最近 `dialogues` 作為對話歷史

### LLM 回應格式擴充

```json
{
  "narrative": "劇情文字",
  "stat_changes": {"hp": -5, "mp": -10, "san": 0},
  "new_items": [{"name": "...", "description": "...", "properties": {...}}],
  "new_npcs": [{"name": "...", "description": "...", "stats": {...}}],
  "dialogues": [{"speaker": "NPC名", "content": "對話內容"}],
  "action_log": "摘要"
}
```

## 全域筆記

### 已知限制

- Discord embed 欄位值限制 1024 字元，長文本需要截斷或多個 embed
- Deepseek API 有 rate limit，高頻 D20 可能會被限制
- 每個論壇頻道最多 5 個標籤

### 未來可擴充方向

- 地點系統（locations 表）
- 戰鬥系統（回合制初始化）
- 任務/Quest 追蹤
- 記憶壓縮（當對話數超過閾值時 AI 摘要）
